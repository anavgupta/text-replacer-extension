MutationObserver & Debounce

Why your extension hammers the CPU on dynamic pages, and a one-function fix that cuts work by 90%.

Problem What happens without debounce

A MutationObserver fires its callback for every single DOM change. On a modern SPA (React, Salesforce Lightning, etc.), a single user action can trigger 50-200 mutations in rapid succession. Without debouncing, our extension runs a full storage read + DOM walk for each one — most of which are wasted since the next mutation immediately invalidates the work.

Solution Debounce: wait, then execute once

A debounce delays execution until mutations stop arriving for a set period (e.g., 150ms). Every new mutation resets the timer. Only the final batch triggers actual work.

content.js — The Fix +6 lines, -1 line
// Debounce helper — delays fn until calls stop for `delay` ms let _debounceTimer = null; function debouncedReplacements(delay = 150) { clearTimeout(_debounceTimer); // Reset timer on every call _debounceTimer = setTimeout(() => performReplacements(), delay); }
// MutationObserver — watch for dynamic content const observer = new MutationObserver(() => { performReplacements(); // OLD: runs on EVERY mutation debouncedReplacements(); // NEW: waits for mutations to settle });

Code How debounce works internally

The entire mechanism is a closure around a single setTimeout ID. Each call clears the previous timer and starts a new one. Only when no new calls arrive within the delay window does the function actually execute.

debounce — step by step
// 1. Outer variable holds the timer ID (persists across calls via closure) let timer = null; function debouncedReplacements() { // 2. Cancel any pending execution from a previous mutation clearTimeout(timer); // 3. Schedule new execution — resets every time a mutation fires timer = setTimeout(() => { // 4. This only runs when no new mutations arrive for 150ms performReplacements(); }, 150); } // Timeline example: // t=0ms Mutation → clearTimeout(null) → setTimeout(fn, 150) → timer pending // t=30ms Mutation → clearTimeout(timer) → setTimeout(fn, 150) → timer RESET // t=55ms Mutation → clearTimeout(timer) → setTimeout(fn, 150) → timer RESET // t=80ms Mutation → clearTimeout(timer) → setTimeout(fn, 150) → timer RESET // t=230ms No mutations for 150ms → fn() EXECUTES → 1 DOM walk
